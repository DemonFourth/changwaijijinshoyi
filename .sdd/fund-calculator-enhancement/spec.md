# 场外基金收益计算器 - 功能增强需求规格

## 1. 概述

### 1.1 项目描述
场外基金收益计算器是一个纯前端Web应用，用于管理场外基金持仓、计算收益（FIFO成本法）。本次增强参考股票收益计算器（gupiaoshouyi-clac）的数据架构和UI布局，对当前项目进行全面改进，重点在于**统一管理模块、方便后期维护**，同时增加主题系统、数据架构和UI布局增强。

**设计原则**：场外基金以长期持有为主，交易频率低，UI和交互设计应体现这一特点——默认按持仓时长排序、图表时间轴适配长周期、避免短期操作导向的功能。

### 1.2 系统范围
- **范围内**：CSS变量设计令牌体系、深色/浅色主题切换、持仓分组、详情页标题栏行情数据、ECharts专业图表、卡片/列表双视图与排序、大数字格式化、交易备注、交易记录分页筛选、Top5盈亏榜单
- **范围外**：后端服务、用户认证、多语言国际化、PWA离线支持、字段显示自定义设置

### 1.3 术语定义
| 术语 | 定义 |
|------|------|
| 设计令牌 | CSS自定义属性，定义颜色、间距、字体等设计基础变量 |
| 持仓分组 | 按持仓状态（持仓中/已清仓）对基金进行自动分组 |
| FIFO | 先进先出成本法，用于计算卖出成本 |
| 大数字格式化 | 将大额数字自动转换为"万"/"亿"单位显示 |

## 2. 功能需求

### REQ-01：CSS变量设计令牌体系与主题切换

**描述**：建立基于CSS自定义属性的设计令牌体系，替代当前硬编码颜色值，并支持深色/浅色主题切换。

**验收标准**：
- **AC-01-01**：当系统加载时，所有颜色、间距、字体、圆角、阴影等样式值应通过CSS自定义属性（`--color-*`、`--spacing-*`、`--font-*`、`--radius-*`、`--shadow-*`）定义，且不包含硬编码颜色值。
- **AC-01-02**：当用户切换主题时，系统应提供"浅色"和"深色"两种主题，切换后所有界面元素的颜色应即时更新，无需刷新页面。
- **AC-01-03**：当用户选择主题后，系统应将主题偏好持久化到LocalStorage，下次打开时自动应用上次选择的主题。
- **AC-01-04**：当系统未检测到用户主题偏好时，应默认使用"浅色"主题。
- **AC-01-05**：当用户操作系统为深色模式时，系统应支持通过`prefers-color-scheme`媒体查询自动适配系统主题。

### REQ-02：持仓分组（持仓中/已清仓自动分组）

**描述**：在汇总页基金列表中，按持仓状态自动将基金分为"持仓中"和"已清仓"两组显示。

**验收标准**：
- **AC-02-01**：当基金列表加载时，系统应根据当前持有份额是否大于0，自动将基金归类为"持仓中"或"已清仓"分组。
- **AC-02-02**：当基金列表显示时，"持仓中"组应排在"已清仓"组之前，每组应有独立的分组标题和基金数量标识。
- **AC-02-03**：当某基金从持仓中变为已清仓（所有份额卖出）时，该基金应自动从"持仓中"组移至"已清仓"组。
- **AC-02-04**：当用户点击分组标题时，应支持折叠/展开该分组下的基金列表。

### REQ-03：详情页标题栏行情数据展示

**描述**：在详情页标题栏中展示基金行情数据（最新净值、估算净值、估算涨幅），替代当前简单的基金名称标题。

**验收标准**：
- **AC-03-01**：当用户进入基金详情页时，标题栏应显示基金名称、基金代码、最新净值、净值日期。
- **AC-03-02**：当存在估算数据时，标题栏应显示估算净值和估算涨幅，估算涨幅应以颜色区分正负（正数绿色、负数红色）。
- **AC-03-03**：当标题栏宽度不足以显示所有数据时，行情数据应自适应换行或收缩，确保基金名称始终可见。

### REQ-04：ECharts专业图表替换

**描述**：使用ECharts图表库替换当前简单的HTML/CSS图表，提供专业的数据可视化。

**验收标准**：
- **AC-04-01**：当汇总页加载时，应显示所有基金的总收益趋势折线图（按持仓周期时间轴）。
- **AC-04-02**：当用户进入详情页时，应显示该基金的收益趋势图、买入卖出对比柱状图、收益率变化折线图。
- **AC-04-03**：当图表数据为空时，应显示"暂无数据"的空状态提示。
- **AC-04-04**：当主题切换时，图表的配色方案应跟随主题自动调整（浅色主题使用浅色背景配色，深色主题使用深色背景配色）。
- **AC-04-05**：当图表容器尺寸变化时（如窗口缩放），图表应自动调整大小以适应容器。

### REQ-05：卡片/列表双视图与排序功能

**描述**：在汇总页基金列表中，支持卡片视图和列表视图两种展示模式，并提供多维度排序功能。默认按收益率降序排列，体现长期持有收益导向。

**验收标准**：
- **AC-05-01**：当用户点击视图切换按钮时，基金列表应在"卡片视图"（方框卡片网格布局）和"列表视图"（紧凑行列表布局）之间切换。
- **AC-05-02**：当用户选择卡片视图时，每个基金应以独立卡片形式展示，包含基金名称、代码、收益率、收益额、最新净值等关键信息。
- **AC-05-03**：当用户选择列表视图时，基金应以表格行形式展示，每行包含相同的字段信息，布局更紧凑。
- **AC-05-04**：当用户点击排序选项时，应支持按收益率、收益额、持仓市值、基金名称进行升序/降序排序，默认按收益率降序。
- **AC-05-05**：当用户切换视图或排序时，视图偏好和排序方式应持久化到LocalStorage。

### REQ-06：大数字格式化（万/亿单位）

**描述**：对大额金额数字进行智能格式化，自动转换为"万"/"亿"单位显示，悬浮显示完整值。

**验收标准**：
- **AC-06-01**：当金额绝对值大于等于10000且小于100000000时，应显示为"X.XX万"格式（如¥12,345.67 → ¥1.23万）。
- **AC-06-02**：当金额绝对值大于等于100000000时，应显示为"X.XX亿"格式（如¥123,456,789.00 → ¥1.23亿）。
- **AC-06-03**：当金额绝对值小于10000时，应保持原有的千分位格式显示（如¥1,234.56）。
- **AC-06-04**：当用户将鼠标悬浮在格式化后的大数字上时，应通过tooltip显示完整的千分位格式金额。
- **AC-06-05**：当大数字格式化应用时，所有统计卡片、基金卡片、详情页中的金额显示均应统一使用该格式化规则。

### REQ-07：交易备注字段

**描述**：为交易记录增加备注字段，支持用户为每笔交易添加文字备注。

**验收标准**：
- **AC-07-01**：当用户添加或编辑交易记录时，表单应包含"备注"输入框，支持输入最多50个字符的文本。
- **AC-07-02**：当交易记录保存时，备注内容应随交易数据一起持久化到LocalStorage。
- **AC-07-03**：当交易记录列表显示时，如有备注内容，应在交易行中显示备注文本（超长截断并悬浮显示全文）。
- **AC-07-04**：当导入/导出数据时，备注字段应包含在数据中，确保数据完整性。

### REQ-08：交易记录分页与筛选

**描述**：增强交易记录表格，支持分页显示和多条件筛选。

**验收标准**：
- **AC-08-01**：当交易记录数量超过每页显示条数（默认10条）时，应显示分页控件，支持翻页和跳转。
- **AC-08-02**：当用户选择每页显示条数时，应支持10/20/50条三种选项。
- **AC-08-03**：当用户使用筛选功能时，应支持按交易类型（买入/卖出/分红）和日期范围进行筛选。
- **AC-08-04**：当筛选条件生效时，应显示筛选结果数量和当前筛选条件标签，支持一键清除筛选。
- **AC-08-05**：当交易记录为空或筛选无结果时，应显示对应的空状态提示。

### REQ-09：Top5盈亏榜单

**描述**：在汇总页增加盈利Top5和亏损Top5基金榜单展示。

**验收标准**：
- **AC-09-01**：当汇总页加载时，应显示"盈利Top5"和"亏损Top5"两个榜单区域。
- **AC-09-02**：当基金数量大于0时，盈利Top5应按收益率从高到低显示前5只收益为正的基金，亏损Top5应按收益率从低到高显示前5只收益为负的基金。
- **AC-09-03**：当榜单中基金数量不足5只时，应显示实际数量的基金，不填充空位。
- **AC-09-04**：当没有盈利/亏损基金时，对应榜单应显示"暂无数据"提示。
- **AC-09-05**：当用户点击榜单中的基金时，应跳转到该基金的详情页。

## 3. 非功能需求

### NFR-01：性能
- 主题切换响应时间应小于100ms。
- ECharts图表首次渲染时间应小于500ms。
- 基金列表排序和视图切换应无感知延迟（小于50ms）。

### NFR-02：兼容性
- 支持Chrome 80+、Firefox 80+、Safari 14+、Edge 80+浏览器。
- ECharts库通过本地lib/echarts.min.js引入，无需CDN依赖。

### NFR-03：可维护性
- CSS变量命名应遵循统一规范（`--{category}-{property}-{variant}`）。
- 新增JS模块应遵循现有模块注册系统（ModuleRegistry），统一管理所有模块。
- 所有新增功能应通过事件总线（EventBus）与现有模块解耦通信。
- 模块职责单一，每个模块只负责一个明确的功能领域，方便后期独立维护和替换。

### NFR-04：数据兼容性
- 新增备注字段应向后兼容，旧数据无备注字段时应显示为空。
- 视图偏好和排序设置应使用独立的LocalStorage键，不影响现有数据结构。
- 大数字格式化为纯展示层逻辑，不改变底层数据存储格式。

## 4. 约束与假设

### 4.1 约束
- 纯前端应用，无后端服务，所有数据存储在LocalStorage中。
- 不引入前端框架（React/Vue等），保持原生JS架构。
- ECharts通过本地lib/echarts.min.js引入，不依赖CDN。

### 4.2 假设
- 用户的LocalStorage空间足够存储所有数据（含新增的备注和设置字段）。
- 基金API（JSONP）接口稳定可用。
- 用户浏览器支持CSS自定义属性（IE不支持，但已在兼容性要求中排除）。
